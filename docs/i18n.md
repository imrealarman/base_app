# Internationalization (i18n)

The template ships with English and Persian translations via
[slang](https://pub.dev/packages/slang) — a code-gen based i18n package that produces a fully
typed `Translations` class, so a typo'd or missing translation key is a compile error instead
of a runtime surprise.

## Where things live

- `lib/i18n/en.i18n.json`, `lib/i18n/fa.i18n.json` — the source translation files. Nested keys
  become nested typed getters (`t.settings.themeMode.system`), and `$variable` placeholders
  become typed function parameters (`t.home.environment(env: config.appEnv)`).
- `build.yaml` — configures `en` as the base/fallback locale, and opts into slang's `legacy`
  builder (writes files directly via `dart:io` instead of build_runner's asset writer, which
  otherwise throws `Asset already exists` if a stale `strings*.g.dart` is already on disk
  without matching `.dart_tool` build-cache metadata — e.g. after switching branches. See
  [code-generation.md](code-generation.md)).
- `lib/i18n/strings.g.dart` (generated, gitignored — regenerate, don't hand-edit) — the
  `AppLocale` enum, `Translations` class, and `context.t` extension.

Regenerate after editing any `.i18n.json` file:

```
dart run build_runner build --delete-conflicting-outputs
```

If you ever see `InvalidOutputException: Asset already exists` from the slang builder, run
`dart run build_runner clean` once and rebuild — it means `.dart_tool`'s build cache fell out
of sync with the files actually on disk.

## Using translations in a widget

```dart
final t = context.t;
Text(t.settings.title)
```

`context.t` rebuilds the widget automatically when the locale changes, as long as a
`TranslationProvider` is an ancestor — `lib/bootstrap.dart` wraps the whole app in one, so this
works everywhere in `lib/features/`.

## Language switching

`lib/features/settings/presentation/widgets/language_tile.dart` lists every `AppLocale` value
as a selectable radio option. Selecting one calls `LocaleSettings.setLocale(locale)` (updates
the running app immediately) and persists the choice via `lib/core/i18n/locale_storage.dart`
(a thin `shared_preferences` wrapper), so the next app launch restores it in
`bootstrap.dart` before falling back to the device locale.

## Adding a new language

1. Create `lib/i18n/<code>.i18n.json` with the same key structure as `en.i18n.json`.
2. Regenerate: `dart run build_runner build --delete-conflicting-outputs`.
3. Add a human-readable label for it in
   `lib/features/settings/presentation/widgets/language_tile.dart`'s `_languageLabels` map.

RTL languages (Arabic, Persian, Hebrew, etc.) don't need any extra setup — `lib/app.dart`
derives `Directionality` automatically from the active locale, so switching to an RTL language
mirrors the layout immediately. See [renaming.md](renaming.md).

## Adding a new translation key

Add it to **every** `.i18n.json` file (missing keys in non-base locales fall back to the base
locale's value at runtime, but every key should still have a real translation before shipping)
and regenerate.
